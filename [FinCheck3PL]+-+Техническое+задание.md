# 2\. Бизнес необходимость

## 2.1 Краткое описание процесса

У коллег из логистики есть процесс проверки документов с данными о заказах, получаемых от СД.  
СД присылают файлы ответственным сотрудникам через разные каналы: почта, Telegram, WhatsApp и др.  
Логист вручную проверяет файл по заданным параметрам. Если находятся ошибки - возвращает файл СД с комментариями.  
СД исправляет файл, и логист проводит повторную проверку.  
После устранения всех ошибок логист загружает файл в Jupyter Notebook, где его дополнительно проверяет скрипт.  
\- Если скрипт находит ошибки - логист отправляет файл обратно в СД с описанием.  
\- Если ошибок нет - логист подтверждает это письмом СД.

## 2.2 Основные проблемы

**Высокая трудоёмкость и зависимость от человека:**

- Проверка вручную занимает много времени, особенно при большом объёме файлов.
- Риск человеческой ошибки (пропуск ошибок, неточности в интерпретации правил).

**Цикличность и задержки из-за повторных проверок:**

- Многократные итерации «СД → логист → СД» увеличивают срок обработки.
- Нет единой точки контроля: сложно отследить статус файла (на каком этапе, кто в работе).

**Отсутствие стандартизации входных данных:**

- Файлы приходят через разные каналы (почта, Telegram, WA) - разный формат, метаданные, имена файлов
- Требуется ручная предобработка (скачивание, переименование, архивация и т.д.).

**Дублирующая проверка:**

- Сначала ручная, потом скриптом в Jupyter → избыточность, если правила одинаковые.
- Скрипт не интегрирован в единый поток: логист сам загружает файл и запускает проверку.

**Нет автоматического уведомления и отчётов:**

- СД получает подтверждение только по email (не гарантировано прочтение).
- Нет логирования: сложно анализировать частоту ошибок, типы дефектов, время цикла.

**Не масштабируемо:**

- При росте объёма документов нагрузка на логистов растёт линейно, а производительность - нет.

## 2.3 Основные цели

- Устранить ручную проверку и минимизировать ошибки
- Сократить долю ручной проверки данных с 100% до 0% путем внедрения автоматической валидаци
- Внедрить шаблоны для каждой СД по загрузки данных, обеспечив 100% соответствие входящих файлов установленной схеме валидации
- Объединить процессы ручной и автоматической верификации в единый пайплайн обработки, высвободив этим 0.8 FTE
- Обеспечить прозрачность и трекинг статусов
- Настроить автоматическую отправку уведомлений (email) при нахождении ошибок в 100% случаев, исключив необходимость ручного информирования логистами
- Высвободить 0.8 FTE  рабочего времени логистов в месяц за счет автоматизации рутинных операций, перенаправив ресурсы на решение нестандартных задач

## 2.4 Потенциальные выгоды

На текущий момент на обработку всех файлов\\листов в месяц логистом затрачивается порядка 0.78 FTE (расчет данного значения приведены ниже). Так же из расчетов следует что для обработки того же количества файлов\\листов в месяц, потребуется использовать 0,00486 лицензий робота.

Также по условиям договоров с партнёрами, если сверка не завершена в установленный срок, компания Авито обязана оплатить всю сумму, указанную контрагентом, без возможности последующего вычета выявленных расхождений.  
Таким образом, скорость и точность сверки напрямую влияют на объём переплат.  
**Оценка эффекта:**  
В текущих условиях своевременно проведённая сверка позволяет ежемесячно избегать переплат на сумму ~13,5 млн руб. по всем партнёрам

# 5\. Критерии приемки робота

## 5.1 DoR (Definition of Ready)

В данном пункте должно быть прописаны требования которые нам необходимы для того что бы начать работу над данным роботом. Такие как:

- Все входные данные предоставлены
- Необходимая и достаточная структура входного\\выходного файлов согласованы
- Учетные записи получены или есть инструкция по получения УЗ
- Тестовые данные предоставлены
- Все Бизнес\\Системные ошибки должны быть прописаны в ТЗ
- Все Позитивные\\Негативные нотификации должны быть прописаны
- Техническое задание написано
- Техническое решение прошло согласование внутри команды
- Техническое задание согласовано
- TDR написан
- TDR согласован
- Задачи в Jira созданы, добавлены в беклог и оценены
- При зависимости от смежной системы необходимы согласованные даты завершения работ
- При критической зависимости от смежной системы\\команды необходимо полное завершение работ

## 5.2 AC (Acceptance Criteria)

В данном пункте должен быть описаны критерии по которому бизнес сможет определить полноту разработанного функционала робота. Такие как: 

- Робот запускается при получении файла из ММ от логиста
- Робот отправляет нотификацию, как при успешном завершении, так и при возникновении ошибок
- Робот получает файл и проводит его верификацию
- При возникновении ошибки робот фиксирует ее и переходить к следующему транзакции
- Робот составляет отчет по результатам работы 

## 5.3 DoD (Definition of Done)

В данном пункте должно быть прописаны требования по которым будет считаться что разработка завершена. Такие как:

- Все части робота разработаны
- Ревью по всем частям робота пройдено
- Анализатор кода пройден, подсвеченные недочеты исправлены
- Робот отрабатывает end-to-end
- Робот покрывает все AC из пункта 5.2
- Все таски по данному роботу закрыты
- Внутренне тестирование пройдено
- Учетные записи для робота получены
- Демо для заказчика проведено
- Все требуемые доступы заказчику предоставлены (сетевая папка, оркестратор)
- Демо для команды проведено
- Бот задеплоен на прод
- Релиз документация написана
- Пользовательская инструкция и инструкция по сопровождению написаны
- Информация о роботе добавлена в таблицу [Реестр разработанных роботов](file:///C:\spaces\RPA\pages\398093009\%25D0%25A0%25D0%25B5%25D0%25B5%25D1%2581%25D1%2582%25D1%2580+%25D1%2580%25D0%25B0%25D0%25B7%25D1%2580%25D0%25B0%25D0%25B1%25D0%25BE%25D1%2582%25D0%25B0%25D0%25BD%25D0%25BD%25D1%258B%25D1%2585+%25D1%2580%25D0%25BE%25D0%25B1%25D0%25BE%25D1%2582%25D0%25BE%25D0%25B2)
- Проведено ревью ТЗ и архитектуры и актуализировано в случае каких либо изменений
- Метрики настроены
- ОПЭ пройдено, согласование получено
